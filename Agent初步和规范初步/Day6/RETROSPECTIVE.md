# Day6 复盘

## 今天完成了什么

今天完成了一个最小可运行的 AI Agent Web Demo：

- 复用 Day5 的知识库检索、召回、重排和引用回答能力
- 搭建了 Node + TypeScript 的最小 Web 服务
- 实现了原生 HTML / CSS / JS 的聊天页面
- 支持回答逐步展示、错误提示、重试按钮和状态切换
- 增加了耗时、成本估算和成功率统计

## 今天最大的收获

最大的收获是：

1. 把 CLI Agent 做成 Web Demo 时，重点不只是“能回答”，还要补上状态、错误和反馈。
2. 流式体验不一定一开始就要接真实 token 流，先用可控的伪流式也能很好地验证产品形态。
3. 可观测性很重要，因为只有看到耗时、成本和成功率，才能判断一个 Demo 是否稳定。
4. 前端产品化不是推倒重做，而是把已有 RAG 核心包装成更容易演示的交互界面。

## 今天踩坑 / 复盘

1. 仓库里之前没有任何前端代码，所以 Day6 不能直接套现成 Web 模板，只能自己搭最小页面。
2. 如果一开始就追求真实 token 流，会把复杂度拉高，所以这版先做伪流式更稳。
3. Web Demo 比 CLI 更容易暴露错误处理问题，因此要把配置错误、索引缺失、请求失败都转成清晰中文提示。

## 当前状态

目前 Day6 已经具备：

- 独立 Web 项目骨架
- 本地知识库与索引构建
- 浏览器聊天页面
- 逐步输出回答

并且已经完成了 `npm install`、`npm run typecheck`、`npm run build-index` 和接口级联调验证。

## 明天进入 Day7 前建议

进入 Day7 前，建议先重点回看 3 件事：

1. 浏览器输入一个问题后，数据是怎么从前端流到后端，再回到前端页面的。
2. 为什么 Day6 要保留 Day5 的 citations 校验，而不是只追求“页面看起来会动”。
3. 如果后面升级成真实 token 流，应该优先改 `model.ts` 和 `server.ts` 哪一层。
