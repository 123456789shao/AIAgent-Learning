# OpenSpec Demo

这个 Demo 演示：**怎么把一个模糊需求写成清晰规范**。

## 场景

需求原话：

> 给聊天页加一个“清空对话”按钮。

如果直接开始写代码，通常会遇到这些问题：

- 按钮放哪里？
- 只清空前端显示，还是也清空服务端状态？
- 要不要二次确认？
- 清空后提示什么？
- 失败时怎么处理？

OpenSpec 的核心就是：**先把需求写清楚，再进入实现。**

---

## 1. 原始需求

```text
给聊天页加一个清空对话按钮。
```

## 2. 规范化后的需求

### 功能目标
用户在聊天页面点击“清空对话”按钮后，可以删除当前会话中的所有消息显示内容，并恢复输入区初始状态。

### 范围
包含：
- 页面增加“清空对话”按钮
- 清空聊天消息列表
- 清空错误提示
- 清空 planner 展示区
- 清空 citations / meta 展示区

不包含：
- 删除服务端日志
- 删除评测记录
- 删除登录状态

### 交互规则
1. 按钮放在聊天输入框附近。
2. 用户点击后弹出二次确认。
3. 用户确认后立即清空页面内当前对话内容。
4. 清空后保留登录状态。
5. 清空后输入框重新获得焦点。

### 异常规则
- 如果当前本来就没有聊天内容，点击后不给报错。
- 如果页面正处于回答生成中，禁止清空，并提示“回答生成中，暂时无法清空对话”。

### 验收标准
- 有聊天记录时，确认后页面消息全部消失。
- planner 区与 meta 区同步恢复默认提示。
- 未登录/已登录都不会因此报错。
- 正在流式输出时不能执行清空。

---

## 3. 这个 Demo 想表达什么

OpenSpec 不是在写代码。

它在做的是：

**把一句模糊话，展开成可实现、可验收、可讨论的规范。**

所以你可以把它理解成：

- 写代码前的需求澄清器
- AI 编程前的输入增强器
- 防止“想的是一回事，做出来是另一回事”的第一道关

---

## 4. 对应到你当前仓库

在你的项目里，它最像：

- 每天开始前先明确“今天到底要做什么”
- 把一句话需求扩展成结构化目标
- 先想清楚边界、异常、验收，再写代码

一句话总结：

**OpenSpec Demo = 演示怎么把一句需求，写成一份能指导实现的规范。**
