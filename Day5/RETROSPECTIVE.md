# Day5 复盘

## 今天完成了什么

今天完成了一个最小可运行的 RAG 问答 Agent：

- 准备了本地 Markdown 知识库
- 把文档切成 chunks
- 为 chunks 生成 embeddings
- 保存本地 `rag-index.json`
- 提问时先检索、再轻量重排、最后回答
- 强制答案附带来源片段

## 今天最大的收获

最大的收获是：

1. RAG 的关键不是“把资料塞给模型”，而是“先检索出最相关的资料，再让模型基于资料回答”。
2. chunk 切得好不好，会直接影响召回质量。
3. 即使模型回答看起来像对，也不代表它真的有依据；引用来源才是更可靠的判断标准。
4. 对学习项目来说，本地 JSON 索引比数据库更容易理解和调试。

## 今天踩坑 / 复盘

1. 如果知识库语料太杂，索引质量会下降，所以 Day5 先只放一小批可控 Markdown 文档。
2. 真正的 reranker 虽然更强，但会引入额外模型和复杂度；当前阶段先用“向量分数 + 关键词匹配”的轻量重排更合适。
3. 如果不强制 citations，模型很容易退回到“看起来像知道”的回答方式。

## 当前状态

## 明天进入 Day6 前建议

进入 Day6 前，建议先彻底验证 3 件事：

1. 检索出来的 chunk 是否真的相关
2. 最终 citations 是否真的能在原文中找到
3. 资料不足时，是否真的会老实说“证据不足”
