# Day7 复盘

## 今天完成了什么

今天把 Day6 的最小 Web Demo 继续推进成了一个更接近真实产品形态的 AI Agent 项目：

- 把单条回答链路拆成了 planner / executor / validator / orchestrator
- 增加了共享密码登录、内存 session 和最小限流
- 补上了 10 条固定评测集和规则化评分逻辑
- 更新了前端页面，支持登录、planner 展示、流式回答、引用和多项指标
- 保持了原生 HTML / CSS / JS + Node + TypeScript 的轻量结构

## 今天最大的收获

最大的收获是：

1. 多 Agent 不一定要做成很重的复杂系统，先把职责拆清楚比“角色名变多”更重要。
2. 一个能上线的最小 Demo，除了回答本身，还必须补上鉴权、限流和失败提示。
3. 评测的价值在于让项目从“能跑”变成“可回归、可比较、可收敛”。
4. 前端体验上先展示 planner，再展示 answer，会让多角色协作更容易被看见和理解。

## 今天踩坑 / 复盘

1. 如果直接沿用 Day6 前端，虽然接口能跑，但用户看不到 Day7 新增的 planner 和鉴权能力，所以前端必须同步改。
2. 评测最好先用规则化打分，不然一开始就上 LLM-as-a-judge，会让结果更难解释也更不稳定。
3. 最小上线能力和完整用户系统不是一回事，Day7 先做共享密码 + 内存 session 已经足够贴合当前学习目标。

## 当前状态

目前 Day7 已经具备：

- 独立 Day7 项目骨架
- 本地知识库与索引构建
- planner / executor / validator 多角色协作
- 登录、登出、鉴权和限流
- 浏览器登录页、聊天页、planner 区和指标区
- 10 条固定评测集与汇总报告输出
- 调用链、执行流程和手动验证文档

当前还差最后一步：

- 安装依赖并完成 `npm run typecheck`、`npm run build-index`、`npm run eval`、`npm run start` 验证

## 下一步建议

接下来建议重点关注 3 件事：

1. planner 的输出是否足够稳定，能不能在同类问题上保持一致的计划结构。
2. validator 对 citations 和证据不足的兜底是否真的能压住幻觉。
3. 如果后面准备上线，应该优先把哪一层先替换成真实持久化能力：session、限流还是评测结果存储。
