# GUI Agent 开发规范

本文档用于约束 `D:\Code\AIAgent-Learning\GUI Agent` 项目的开发方式，保证这个项目在从 0 到 1 的过程中保持清晰、可控、可扩展。

## 1. 总体原则

### 1）先做最小可运行版本

第一阶段优先目标不是“能力最强”，而是：

- 能跑通
- 能观察
- 能验证业务价值
- 能快速迭代

不要一开始就追求：

- 通用万能 GUI Agent
- 全自动跨系统桌面机器人
- 大而全的平台化系统

### 2）先提示，后执行

当前项目优先做：

- 页面识别
- 关键节点提醒
- 下一步建议

自动点击、自动输入、自动滚动等执行能力，必须放在后续阶段逐步开放。

### 3）先规则闭环，后模型增强

第一阶段优先保证：

- 感知结果可解释
- 提示逻辑可验证
- 页面识别可调试
- 决策结果可追踪

不要一开始把所有逻辑都丢给大模型。

### 4）先单场景，后多场景

开发时应优先选一个清晰业务场景打透，例如：

- 某个固定后台页面流程
- 某个固定录入场景
- 某个固定审核流程

不要第一版同时支持很多系统、很多页面、很多任务。

### 5）可观察性优先

系统必须尽量让开发者和用户看得见：

- 当前识别到了什么
- 当前命中了什么规则
- 当前为什么提醒
- 当前准备执行什么动作
- 当前执行后结果如何

不做黑盒系统。

---

## 2. 产品范围规范

### 第一阶段允许做的内容

- 截图与当前窗口识别
- OCR 文本识别
- 页面状态判断
- 文档规则匹配
- 弹窗提醒 / 悬浮提示 / 侧边栏提示
- 下一步建议输出
- 基础 trace 输出

### 第一阶段不建议做的内容

- 全自动跨系统操作
- 无确认直接点击/输入
- 同时支持多个复杂业务域
- 上来就做多 Agent 协作
- 上来就做复杂平台配置系统

### 自动执行的约束

如果后续加入自动执行，必须满足：

- 默认需要用户确认
- 必须有明确执行目标
- 必须能记录执行动作
- 高风险动作必须额外确认

高风险动作包括但不限于：

- 提交
- 删除
- 覆盖
- 审核通过
- 对正式数据产生影响的操作

---

## 3. 架构规范

项目架构建议始终围绕以下核心分层：

- Perceptor 感知层
- Knowledge / Memory 知识层
- Planner / Decision 决策层
- Operator 交互层
- State Loop 状态循环

### 分层职责要求

#### 1）Perceptor 只负责感知

负责：

- 截图
- OCR
- 当前窗口信息
- 页面特征提取

不负责：

- 业务判断
- 复杂提示生成
- 自动执行决策

#### 2）Knowledge 只负责知识与规则

负责：

- SOP
- 页面规则
- 提示模板
- 风险规则
- 当前业务说明

不负责：

- 直接控制界面
- 直接做截图/OCR

#### 3）Decision 只负责判断当前动作

负责：

- 是否提醒
- 提醒内容
- 是否建议下一步
- 是否允许执行动作

不负责：

- 直接执行点击
- 直接存储原始截图

#### 4）Operator 只负责展示或执行

负责：

- 弹窗
- 悬浮提示
- 高亮
- 点击/输入/滚动（后续）

不负责：

- 业务规则判断
- 页面语义理解

### 状态流规范

推荐所有功能都按以下最小闭环设计：

**感知 → 理解 → 匹配 → 决策 → 提示/执行 → 再感知**

不要出现大量跨层直接调用，避免后期架构混乱。

## 4. 技术选型规范

### 语言

- 默认使用 Python
- 没有充分理由，不引入第二主语言

### GUI 感知

- 截图优先使用 `mss`
- 图像处理可配合 `Pillow`
- OCR 优先考虑 `PaddleOCR`

### 自动化执行

- 最小原型优先使用 `pyautogui`
- 后续如果需要更强控制，再评估其他方案

### 模型使用

- GUI 理解优先考虑多模态模型
- 如果只是文字识别 + 规则匹配，可先用 OCR + 文本模型
- 第一阶段不要过度依赖纯模型黑盒推理

### UI

- 最小版本可先用 `Tkinter`
- 如果要更像正式桌面产品，再升级到 `PySide6`

### 文档知识

- 第一阶段优先本地 markdown / json / yaml
- 不强制一开始上向量库或复杂 RAG

## 5. 代码规范

### 1）优先保持简单直接

- 不做过度抽象
- 不为未来假设场景提前设计复杂框架
- 一个阶段只解决一个阶段的问题

### 2）命名要清晰

命名应直接表达职责，例如：

- `screen_capture`
- `ocr_reader`
- `page_detector`
- `rule_matcher`
- `decision_engine`
- `overlay_notifier`

避免含义模糊的命名，例如：

- `manager`
- `helper`
- `utils2`
- `temp_handler`

### 3）配置和逻辑分离

需要可调的内容应尽量外置：

- 页面规则
- 提示文案
- 风险动作阈值
- 采样频率
- 场景配置

不要把这些内容全部写死在业务逻辑里。

### 4）优先可读性

这个项目第一阶段是原型和验证项目，代码要便于理解和调试。

优先：

- 清晰
- 可读
- 可改

而不是过度追求抽象层数。

---

## 6. 日志与可观测规范

系统至少应能输出以下信息：

- 当前时间
- 当前活动窗口
- 当前识别到的页面类型
- 当前命中的规则
- 当前决策结果
- 当前提示内容
- 当前执行动作（如果有）
- 当前执行结果

### 建议保留的 trace 字段

- `window_name`
- `page_type`
- `ocr_text`
- `matched_rule`
- `decision_type`
- `decision_reason`
- `operator_action`
- `operator_result`
- `step_id`
- `timestamp`

### 规范要求

- 关键流程必须可追踪
- 用户看到的提醒，最好能对应到内部决策原因
- 后续若出现误提醒、误操作，必须能回溯原因

---

## 7. 安全与风险控制规范

GUI Agent 和普通文本 Agent 不同，风险更高，因为它可能直接作用到真实界面。

因此必须遵守以下规范。

### 1）默认保守

- 第一阶段默认只提醒，不自动执行
- 所有执行能力默认关闭
- 必须由明确配置或用户确认后开启

### 2）高风险动作必须确认

如下动作不得默认自动执行：

- 提交
- 删除
- 审批
- 覆盖
- 支付
- 发送
- 任何影响正式业务数据的动作

### 3）执行前后必须留痕

如果执行了任何动作，至少要记录：

- 执行动作类型
- 执行时间
- 执行目标
- 执行前判断依据
- 执行结果

### 4）异常时优先停止

如果出现以下情况，应优先停止而不是继续自动操作：

- 页面识别不明确
- OCR 结果异常
- 页面结构明显变化
- 目标按钮/输入框定位不稳定
- 用户环境发生变化

---

## 8. 业务规则规范

### 1）业务文档要结构化

不要把所有 SOP 只当自然语言大段文本。

应尽量结构化为：

- 页面名称
- 页面特征
- 当前步骤
- 提醒条件
- 提醒文案
- 下一步建议
- 风险动作说明

### 2）规则优先小而清晰

第一阶段规则不要过大而全。

推荐：

- 一个页面一组规则
- 一个关键节点一个提醒
- 一个风险点一个检查项

### 3）文案要短、直接、可执行

提醒文案要尽量像操作助手，而不是长篇说明文。

例如：

- “当前缺少手机号，请先补齐后再提交。”
- “你已进入审核页，下一步先核对订单金额。”
- “这是正式提交前的确认框，请再次确认信息无误。”

---

## 9. 开发流程规范

建议按下面顺序推进：

### 第一阶段

- 选定一个场景
- 明确页面
- 识别截图/OCR
- 建规则
- 弹提醒

### 第二阶段

- 提升页面识别准确率
- 增加状态跟踪
- 优化提示时机
- 补 trace

### 第三阶段

- 加用户确认执行
- 加基础点击/输入
- 做动作回放和日志

### 第四阶段

- 扩展更多场景
- 引入更强模型
- 增加更复杂工作流
- 再考虑平台化能力

---

## 10. 一句话规范总结

这个项目必须始终坚持：

**先做最小可运行、可观察、低风险的 GUI 提示助手，再逐步向半自动和自动执行演进。**

不要一开始就追求全自动、全场景、全平台。

先把一个场景做稳，比做一个很大的空架子更重要。
