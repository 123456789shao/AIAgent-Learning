# GUI Agent 项目文档

本文档用于说明 `D:\Code\AIAgent-Learning\GUI Agent` 项目的方向，包括产品定位、业务场景、业务价值、整体架构、技术选型和后续演进路线。

## 1. 项目定位

这个项目不建议一开始就做成“全自动桌面操作机器人”，而更适合先做成一个：

**基于 GUI 感知、业务文档和大模型理解能力的实时提示助手。**

它的核心目标不是立刻替代用户完成所有操作，而是：

- 看懂用户当前处在什么页面
- 识别当前页面在做什么
- 结合预先提供的业务文档或 SOP
- 在关键节点给出提醒、建议和下一步指引
- 后续再逐步升级到半自动、再到自动执行

所以它的第一阶段定位，更像：

- GUI 提示助手
- 操作辅助助手
- 桌面智能引导系统
- 带界面理解能力的业务流程助手

---

## 2. 为什么要单独开一个项目

这个 GUI Agent 项目和现有的 `langchain.py学习` 项目虽然都属于 Agent 范畴，但本质上不是一类系统。

当前已有项目更偏：

- 文本型 Agent
- tool calling
- workflow / loop / ReAct
- 前后端演示
- 学习型原型

而 GUI Agent 更偏：

- 屏幕感知
- OCR / 图像理解
- 桌面交互
- 键鼠控制
- 页面状态识别
- 视觉 + 决策 + 操作闭环

因此单独开项目的好处是：

- 技术边界更清晰
- 不会把现有学习项目搞乱
- GUI Agent 的依赖、结构、测试方式都能独立演进
- 后续更容易沉淀成一个独立方向

所以 `D:\Code\AIAgent-Learning\GUI Agent` 作为独立项目是合理的。

## 3. 产品形态

## 当前最推荐的产品形态

建议先做成：

**“会看当前界面，并在关键时机给提示的 GUI 提示助手”**

它的基础交互形态可以是：

- 用户正常操作业务系统
- Agent 在后台持续感知当前屏幕
- 当识别到特定页面、特定字段或特定状态时
- 在侧边栏、悬浮窗或弹窗里给出提醒

例如：

- “你现在进入了订单审核页面，下一步应该先核对客户信息。”
- “当前页面缺少必填字段，建议先补充手机号。”
- “你现在已经到了第 3 步，下一步应该点击提交审核。”
- “当前弹窗是确认框，提交后会影响正式数据，请再次确认。”

这种形态的优势是：

- 风险低
- 可快速验证
- 用户更容易接受
- 更适合从 0 到 1 落地

---

## 后续可能扩展的产品形态

随着能力增强，它还可以继续发展成：

### 1）半自动 GUI 助手

- Agent 先给建议
- 用户确认后执行点击、输入、滚动
- 保留人工确认环节

### 2）流程型 GUI Agent

- Agent 不只是提示一步，而是跟踪整个任务流程
- 能识别“当前做到哪一步”
- 能做连续多步协助

### 3）桌面级业务操作助手

- 可跨系统工作
- 可操作桌面应用 + Web 应用
- 具备更强的跨页面流程能力

### 4）行业场景专属 GUI Agent

例如：

- 客服工作台助手
- ERP / CRM 操作助手
- 财务录入助手
- 运营后台辅助助手
- 数据填报辅助助手

## 4. 业务场景

这个项目最适合落地在“流程明确、页面固定、用户容易出错、培训成本高”的业务场景。

## 场景一：新员工上手辅助

很多内部系统功能复杂、字段多、流程长，新人需要反复看文档。

GUI Agent 可以：

- 在操作过程中识别当前页面
- 动态给出当前步骤说明
- 告诉用户下一步该做什么

价值：

- 降低培训成本
- 减少带教压力
- 缩短上手时间

---

## 场景二：关键节点防错提醒

一些业务系统在提交、审核、填写时很容易漏项或误操作。

GUI Agent 可以：

- 识别关键页面
- 在提交前提醒检查重点字段
- 在高风险动作前提醒用户确认

价值：

- 降低出错率
- 减少返工
- 提升流程准确性

---

## 场景三：复杂流程辅助

当一个业务流程跨多个页面、多次点击、多次输入时，用户很容易忘记步骤。

GUI Agent 可以：

- 识别当前流程节点
- 提示下一步动作
- 跟踪是否已完成关键步骤

价值：

- 降低认知负担
- 提高操作效率
- 提升流程完成率

---

## 场景四：标准作业指导（SOP）落地

很多公司已经有 SOP 文档，但真正执行时大家不会一直看文档。

GUI Agent 可以把 SOP 从“静态文档”变成“实时提示”。

价值：

- 让 SOP 真正进入操作现场
- 提高执行一致性
- 把文档价值转成实际效果

---

## 场景五：半自动业务助手

当系统能力进一步增强后，GUI Agent 可以在用户确认下帮助做部分重复性操作。

比如：

- 自动点击固定按钮
- 自动填写固定字段
- 自动切换页面
- 自动执行简单的重复流程

价值：

- 进一步提效
- 把部分重复操作自动化
- 从“提示助手”升级成“执行助手”

---

## 5. 业务价值

这个项目的业务价值可以分成当前阶段价值和后续发展价值。

## 当前阶段价值

### 1）培训价值

它可以把业务文档和操作说明直接嵌入用户真实操作场景中。

相比传统培训方式，它更接近“边做边学”。

### 2）防错价值

它能在关键页面和关键动作前给提醒，从而减少漏填、误点和跳步骤。

### 3）提效价值

即使第一阶段只是提示，不做自动点击，也已经能减少用户频繁查文档、来回确认流程的时间。

### 4）验证价值

从技术角度，它也能验证以下能力是否可行：

- 屏幕感知是否稳定
- OCR 是否够用
- 页面识别是否足够准确
- 文档知识能否和 GUI 场景结合
- 提示时机是否合理

---

## 后续发展价值

### 1）流程自动化价值

一旦加入半自动/自动操作能力，它就会从“提示系统”升级为“执行系统”。

### 2）知识沉淀价值

企业原本分散在文档、人员经验和口头培训里的知识，可以转成结构化规则与 Agent 能力。

### 3）业务复制价值

一旦在一个场景跑通，后面可以很快复制到其他业务系统或流程中。

### 4）平台化价值

如果后续继续做强，它甚至可以变成一个“企业 GUI Agent 平台”的雏形：

- 接不同业务文档
- 配不同页面规则
- 支持不同系统
- 输出统一提示/执行能力

---

## 6. 整体架构思路

建议先采用 **4 层架构 + 1 个状态控制流**。

## 6.1 架构总览图

```text
┌──────────────────────────────────────────────┐
│               Environment / GUI             │
│   桌面应用、Web 页面、弹窗、表单、按钮、状态   │
└─────────────────────┬────────────────────────┘
                      │
                      ▼
┌──────────────────────────────────────────────┐
│            1) Perceptor 感知层               │
│ 截图 / OCR / 当前窗口识别 / 页面状态识别      │
└─────────────────────┬────────────────────────┘
                      │ 感知结果
                      ▼
┌──────────────────────────────────────────────┐
│         2) Knowledge / Memory 知识层         │
│ SOP / 操作手册 / 页面规则 / 风险提醒 / 历史状态 │
└─────────────────────┬────────────────────────┘
                      │ 规则匹配 / 上下文补充
                      ▼
┌──────────────────────────────────────────────┐
│        3) Planner / Decision 决策层          │
│ 判断当前节点 / 是否提醒 / 提醒内容 / 是否执行 │
└─────────────────────┬────────────────────────┘
                      │ 决策结果
                      ▼
┌──────────────────────────────────────────────┐
│            4) Operator 交互层                │
│ 悬浮提醒 / 弹窗 / 高亮 / 点击 / 输入 / 滚动   │
└─────────────────────┬────────────────────────┘
                      │ 执行反馈
                      ▼
┌──────────────────────────────────────────────┐
│              State Loop 状态循环             │
│ 继续观察 → 校验结果 → 进入下一轮感知           │
└──────────────────────────────────────────────┘
```

这个架构的核心不是“一次性做完整个任务”，而是让系统持续执行一个最小闭环：

**感知 → 理解 → 匹配知识 → 生成决策 → 提示/执行 → 再感知**

这也是 GUI Agent 最适合从 0 到 1 的架构方式。

## 6.2 架构层之间怎么配合

### 第一步：Perceptor 先看见当前界面

它负责把原始 GUI 环境转成机器可理解的信息，例如：

- 当前活动窗口是什么
- 当前截图里有哪些关键文本
- 当前是否出现弹窗
- 当前页面更像哪个业务页面
- 当前可能处在流程的第几步

这一层不负责决定怎么做，它只负责“把界面事实提取出来”。

### 第二步：Knowledge / Memory 提供业务解释

感知层拿到的是“页面上看到了什么”，但还不知道“这意味着什么”。

知识层负责提供：

- 这个页面通常属于哪个业务环节
- 到这个页面应该提醒什么
- 哪些字段是关键字段
- 哪些状态属于高风险或异常
- 下一步标准操作是什么

也就是说，知识层负责把“界面事实”翻译成“业务语义”。

### 第三步：Planner / Decision 决定当前动作

决策层综合：

- 当前感知结果
- 文档/SOP/规则
- 当前会话状态

然后输出一个动作决策。第一阶段推荐只保留三类：

- `no_action`：当前不用提醒
- `remind`：需要提醒用户
- `suggest_next_step`：明确告诉用户下一步

后面再扩展成：

- `click`
- `type`
- `scroll`
- `wait`

### 第四步：Operator 把决策作用到界面

交互层负责把决策变成可见结果：

- 弹一个提示框
- 在侧边栏展示建议
- 高亮某个区域
- 或者在后续版本里代用户执行点击/输入

这一步是用户真正能感知到的产品输出。

### 第五步：进入 State Loop

无论是提醒还是执行，系统都不应该结束，而是继续观察：

- 用户是否按提示做了
- 页面是否发生变化了
- 是否进入下一步了
- 是否出现异常状态了

所以 GUI Agent 的核心不是单次问答，而是**持续观察下的状态循环系统**。

## 6.3 推荐的最小控制流

第一阶段建议把控制流保持得非常简单：

```text
截图 → OCR/页面识别 → 命中文档规则 → 生成提醒 → 展示提醒 → 等待下一轮截图
```

这个版本适合先做“提示助手”。

后续再升级为：

```text
截图 → 页面理解 → 状态判断 → 决策 → 提示/执行 → 校验结果 → 下一轮
```

这个版本才开始接近真正的 GUI Agent。

## 6.4 为什么这个架构适合当前阶段

因为你现在最需要的不是一个大而全的自动化平台，而是一个：

- 能快速落地
- 能最小闭环
- 能验证业务价值
- 能逐步升级

的架构。

这套 **4 层 + 状态循环** 的方式有几个优点：

- 分层清晰，不容易混乱
- 先做提醒，再做执行，风险低
- 后面可以很自然地接入更强模型
- 后面也能继续演进成更复杂的 workflow / LangGraph 架构

所以这套架构既适合当前 MVP，也适合后续扩展。

## 1）Perceptor 感知层

负责“看当前界面发生了什么”。

主要职责：

- 截图
- OCR 识别
- GUI 元素分析
- 页面状态判断
- 当前窗口识别
- 检测关键字、关键区域、关键弹窗

输入：

- 当前屏幕
- 当前活动窗口
- 当前页面截图

输出：

- 当前页面文本信息
- 关键视觉元素
- 页面类型判断
- 当前可能所处的流程阶段

这是 GUI Agent 的“眼睛”。

---

## 2）Knowledge / Memory 知识层

负责“知道业务规则和文档内容”。

内容来源包括：

- SOP 文档
- 操作手册
- 页面说明
- 业务规则
- 常见错误说明
- 关键提醒清单

它的作用不是单纯存文档，而是给决策层提供：

- 当前页面该提醒什么
- 当前阶段的目标是什么
- 什么情况属于异常/高风险
- 下一步标准动作是什么

这是 GUI Agent 的“知识库”。

---

## 3）Planner / Decision 决策层

负责“基于当前界面和知识，决定现在该做什么”。

它需要判断：

- 当前处在什么页面 / 什么流程节点
- 当前是否需要提醒
- 如果提醒，提醒内容是什么
- 当前是否可以自动执行动作
- 如果可以执行，应该执行什么动作

这是 GUI Agent 的“大脑”。

第一阶段建议它只输出：

- no_action
- remind
- suggest_next_step

第二阶段再增加：

- click
- type
- scroll
- wait

---

## 4）Operator 交互层

负责“把决策真正作用到用户界面上”。

第一阶段主要做：

- 悬浮提示
- 侧边栏提示
- 弹窗提醒
- 步骤指引

后续可扩展：

- 鼠标点击
- 键盘输入
- 滚动
- 快捷键触发
- 焦点切换

这是 GUI Agent 的“手”。

## 7. 技术选型

这个项目建议遵循一个原则：

**先用最简单、最容易跑通的技术栈做最小版。**

## 语言

### 推荐：Python

原因：

- GUI 自动化生态成熟
- OCR、截图、桌面控制工具丰富
- 适合快速原型
- 后续接模型也方便

---

## 屏幕截图 / 屏幕采集

### 推荐：mss

作用：

- 快速截屏
- 支持高频采样
- 比较适合桌面感知

也可以配合：

- Pillow

---

## 键鼠控制

### 推荐：pyautogui

作用：

- 点击
- 输入
- 移动鼠标
- 滚动
- 模拟键盘

优点：

- 上手快
- 跨平台
- 足够支撑最小 GUI Agent 原型

---

## OCR

### 推荐方案一：PaddleOCR

优点：

- 中文支持更好
- 适合业务系统中文界面

### 推荐方案二：Tesseract

优点：

- 经典方案
- 生态成熟

建议：

- 如果界面以中文为主，优先 PaddleOCR

---

## 大模型

### 推荐：支持多模态 GUI 理解的模型

原因：

- GUI 理解不只是读文字
- 还要看布局、按钮、输入框、图标、弹窗
- 纯文本模型只能吃 OCR 结果，能力有限

建议方向：

- 如果想做更强 GUI 理解：选多模态模型
- 如果只是先做“文字识别 + 规则提醒”：可先用 OCR + 文本模型

也就是说：

- **真正的 GUI 理解，最好用多模态模型**
- **最小原型可以先不一步到位**

---

## 桌面 UI

### 最小版推荐：Tkinter

优点：

- Python 自带
- 适合快速做悬浮窗/提示框

### 稍微产品化推荐：PySide6

优点：

- UI 能力更强
- 更适合做桌面悬浮助手
- 视觉上更像正式产品

建议：

- 第一版用 Tkinter 够了
- 想做更像产品再换 PySide6

---

## 文档知识层

### 第一阶段推荐：本地 markdown / json / yaml

不要一开始上复杂向量库。

先把：

- SOP
- 页面规则
- 提示文案
- 场景说明

放成结构化本地文件即可。

后续再升级为：

- RAG
- 向量检索
- 企业知识库接入

---

## 工作流/Agent 编排

### 第一阶段推荐：简单循环 + 状态机

当前不一定需要上很重的框架。

先做：

- 感知
- 判断
- 提示
- 等待下一轮感知

就够了。

### 后续推荐：LangGraph 或自定义状态流

当系统开始出现：

- 多步骤流程
- 多类型动作
- 回退/重试
- 多节点决策

再考虑上 LangGraph 会更合适。

---

## 8. 推荐的最小落地路线

### 第一阶段：看懂并提醒

目标：

- 能识别当前页面或关键文本
- 能匹配文档规则
- 能在合适时机弹出提示

这是最值得先做的版本。

### 第二阶段：用户确认后执行

目标：

- 提示后增加“帮我执行”按钮
- 在确认后执行点击、输入等操作

### 第三阶段：流程跟踪

目标：

- 跟踪用户处在哪一步
- 动态判断下一步
- 识别异常状态和漏步骤

### 第四阶段：更强的 GUI Agent

目标：

- 引入更强多模态理解
- 支持复杂操作
- 支持更多系统和页面
- 支持业务配置化

---

## 9. 当前最推荐的方向

如果从 0 到 1，只做一个最合适版本，我建议是：

**“基于业务文档 + 屏幕识别 + 实时提示的 GUI 提示助手”**

不要一开始就做：

- 全自动桌面机器人
- 通用万能 GUI Agent
- 多系统全流程自动操作平台

因为那样复杂度和风险都太高。

先做“会看、会判断、会提醒”，这是最稳的切入点。

---

## 10. 一句话总结

这个 GUI Agent 项目最适合先做成：

**一个能理解当前界面，并结合业务文档在关键时机给出提醒和操作建议的桌面提示助手。**

它当前阶段最核心的业务价值是：
