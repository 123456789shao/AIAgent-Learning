# 开发规范

适用范围：`D:\Code\AIAgent-Learning`

## 1. 总体目标

本仓库用于学习和实践 AI Agent 的核心能力，代码应优先满足以下目标：

1. **易懂**：方便初学时阅读和复盘。
2. **可运行**：每个 Day 目录都应能独立安装、独立启动、独立验证。
3. **可扩展**：后续 Day4 ~ Day7 可以在前面基础上继续演进。
4. **最小化实现**：只做当前学习目标需要的内容，不过度设计。

---

## 2. 目录规范

### 2.1 Day 目录独立

每一天的练习目录建议独立维护，例如：

- `Day1/`
- `Day2/`
- `Day3/`

每个目录原则上应包含：

- `package.json`
- `tsconfig.json`
- `.env.example`
- `.gitignore`
- `README.md`
- `TASKS.md`
- `RETROSPECTIVE.md`
- `src/`

### 2.2 src 目录职责清晰

推荐按职责拆分：

- `src/index.ts`：CLI 入口，负责串主流程
- `src/config.ts`：读取环境变量与运行配置
- `src/model.ts`：封装模型调用
- `src/prompts.ts`：集中管理 Prompt 构造逻辑
- `src/schema.ts`：定义结构化输出与校验规则
- `src/tools.ts`：定义工具及工具元信息
- `src/agent.ts`：负责编排流程
- `src/retry.ts`：负责重试逻辑（如果当天目标需要）

要求：

- 一个文件只负责一类事情。
- 不要把“模型调用、工具执行、日志打印、校验逻辑”全部混在一个文件里。

## 3. 代码风格规范

## 3.1 基本原则

1. 优先写**简单直接**的实现。
2. 优先写**初学者能看懂**的代码。
3. 不为了“高级感”引入复杂抽象。
4. 能用 20 行讲清楚的逻辑，不拆成 5 层封装。

## 3.2 命名规范

- 文件名使用小写英文，按模块职责命名：
  - `config.ts`
  - `model.ts`
  - `prompts.ts`
  - `schema.ts`
- 函数名使用动词开头，表达动作：
  - `getConfig`
  - `askModel`
  - `runAgent`
  - `parseFinalResult`
- 类型名使用 PascalCase：
  - `Config`
  - `FinalResult`
  - `AgentStep`

## 3.3 注释规范

注释以“帮助理解”为目标，不写废话。

推荐写注释的地方：

- 文件职责说明
- 关键函数作用
- 容易混淆的编排流程
- 为什么要这样做，而不是只写做了什么

不推荐写注释的地方：

- 代码本身已经很清楚的简单赋值
- 与代码重复的空洞说明

示例：

```ts
// 串起 Day3 主流程：决策动作、执行工具、生成最终答案。
export async function runAgent() {}
```

## 4. 模块职责规范

## 4.1 index.ts

只负责：

- 读取命令行参数
- 获取配置
- 调用主流程
- 打印结果
- 统一错误处理

不要在 `index.ts` 里直接写：

- 模型调用细节
- 工具执行细节
- Schema 校验细节

## 4.2 model.ts

只负责模型调用。

要求：

- 输入参数明确
- 返回原始文本或约定结果
- 不承担业务编排
- 不承担工具执行

## 4.3 prompts.ts

只负责 Prompt 相关逻辑。

要求：

- Prompt 文本集中维护
- 用函数构造动态 Prompt
- 明确系统提示词和用户提示词的职责

## 4.4 schema.ts

只负责：

- 定义结构化数据 Schema
- 解析模型输出
- 做运行时校验

要求：

- 尽量用 Zod 做严格约束
- 出错信息要尽量清晰
- 不把工具执行逻辑写进这里

## 4.5 tools.ts

只负责：

- 定义工具名称
- 定义工具输入
- 定义工具元数据
- 执行工具分发

要求：

- 工具输入要尽量窄
- 工具能力要具体，不要过于泛化
- Day3 阶段可以先用 mock 工具，不必一开始接真实 API

## 4.6 agent.ts / retry.ts

负责编排。

例如：

- 调模型
- 解析动作
- 判断是否调用工具
- 执行工具
- 回传工具结果
- 重试失败请求
- 记录执行轨迹

要求：

- 编排层负责流程控制
- 校验失败、工具失败、模型失败要在这里被串起来处理

---

## 5. Prompt 规范

1. Prompt 要写清楚约束，不要只写模糊要求。
2. 需要结构化输出时，必须明确：
   - 只能返回 JSON
   - 字段名是什么
   - 字段类型是什么
   - 不允许输出解释文字
3. 需要工具调用时，必须明确：
   - 有哪些工具
   - 每个工具适合什么场景
   - 每个工具允许什么参数
   - 不允许编造工具
4. Prompt 以“稳定可控”为目标，而不是只追求“聪明回答”。

---

## 6. 错误处理规范

1. 所有 CLI 项目都要有统一错误处理。
2. 报错信息优先使用中文，方便学习时快速定位。
3. 对外报错要具体，避免只输出模糊信息。

示例：

- `缺少 GEMINI_API_KEY。请先复制 .env.example 为 .env 并填写你的密钥。`
- `模型动作不是合法 JSON`
- `summarizeWebpage 缺少 url 参数`

要求：

- 不吞错误
- 不静默失败
- 能明确指出是配置问题、模型输出问题，还是工具参数问题

---

## 7. 环境变量规范

1. 敏感信息只放本地 `.env`。
2. 仓库内只提交 `.env.example`，不提交真实 `.env`。
3. 默认环境变量命名统一：
   - `GEMINI_API_KEY`
   - `GEMINI_MODEL`
4. `.gitignore` 必须忽略：
   - `node_modules/`
   - `.env`

要求：

- 不在代码里硬编码真实密钥
- 不在聊天、文档、截图里暴露真实 key

---

## 8. 文档规范

每个 Day 目录至少维护 3 个文档：

## 8.1 README.md

必须说明：

- 这个 Day 的目标
- 项目做了什么
- 如何安装依赖
- 如何配置 `.env`
- 如何运行
- 调用链/执行链说明
- 示例输出

## 8.2 TASKS.md

用于记录当天任务清单，建议使用 checklist：

```md
- [x] 初始化项目
- [x] 接入模型调用
- [x] 增加结构化输出校验
```

## 8.3 RETROSPECTIVE.md

用于记录复盘，建议包含：

- 今天完成了什么
- 学到了什么
- 遇到了什么问题
- 下一步适合做什么

---

## 9. 开发流程规范

建议遵循以下顺序：

1. 先明确当天目标
2. 先搭最小可运行骨架
3. 再补模块拆分
4. 再补校验与错误处理
5. 再补 README / TASKS / RETROSPECTIVE
6. 最后运行验证

要求：

- 每完成一个 Day，都要保证能本地独立运行
- 每完成一个核心阶段，都要能解释清楚调用链
- 学习项目优先保证“能讲明白”

---

## 10. 验证规范

每个 Day 完成后至少做以下检查：

1. `npm install`
2. `npm run typecheck`
3. `npm run ask -- "你的问题"`
4. 检查终端输出是否符合预期
5. 检查 README 是否与代码一致

如果项目包含结构化输出或工具调用，还要额外验证：

- 非法 JSON 的处理是否清晰
- 参数缺失时是否能报错
- 工具调用轨迹是否可见
- 最终结果结构是否稳定

---

## 11. Git 提交规范

1. 一个 Day 的主要学习成果尽量形成独立提交。
2. commit message 要清晰表达“这次完成了什么”。
3. 推荐格式：

```bash
feat: 完成 Day3 Tool Calling 入门项目，实现工具决策、工具执行与执行轨迹展示
```

常用前缀：

- `feat:` 新功能
- `fix:` 修复问题
- `docs:` 文档更新
- `refactor:` 重构
- `chore:` 杂项维护

---

## 12. 当前仓库的实践方向

结合当前学习路线，推荐按下面节奏推进：

- **Day1**：学会调用模型，理解 Prompt 如何影响输出
- **Day2**：学会约束输出，理解结构化输出与校验
- **Day3**：学会工具调用，理解“模型决策，代码执行”
- **Day4**：学会加入记忆
- **Day5**：学会接入外部数据（RAG）
- **Day6**：做成可演示应用
- **Day7**：总结并形成自己的最小 Agent 框架认知

---

## 13. 一句话原则

这个仓库的开发原则只有一句话：

**先做最小可运行版本，再逐步把 LLM、规范、框架三部分拆清楚。**
